# Ручное тестирование BuhBot: SLA, классификация, callback-кнопки

> **Дата последнего обновления:** 30.03.2026
> **Версия:** v0.30.2+ (PRs #281-#286)
> **Окружение:** Production `https://buhbot.aidevteam.ru`
> **Цель:** полная ручная проверка SLA-системы, AI-классификации, callback-действий и веб-интерфейса

---

## 1. Что именно проверяем

В этой проверке нужно подтвердить:

**SLA и уведомления:**
1. SLA breach-уведомления приходят нужным получателям.
2. Менеджер и администратор видят корректную клавиатуру в Telegram.
3. Бухгалтер видит свою клавиатуру без кнопки **«Уведомить бухгалтера»**.
4. Кнопка **«Уведомить бухгалтера»** реально доставляет уведомление бухгалтеру.
5. При нескольких бухгалтерах уведомление уходит всем настроенным получателям.
6. Кнопка **«Отметить решённым»** закрывает заявку как `closed` (не `answered`).
7. Старые кнопки в неактуальных alert-сообщениях не ломают состояние.
8. Ручной `resolve` не искажает статистику бухгалтера.

**Классификация сообщений (PRs #281, #284):**
9. AI-классификатор (`xiaomi/mimo-v2-flash`) корректно различает REQUEST, SPAM, GRATITUDE, CLARIFICATION.
10. При недоступности AI — fallback на вторую модель (`google/gemini-2.0-flash-001`), затем keyword classifier.
11. При полном отказе AI — Telegram-алерт приходит менеджерам.
12. Keyword fallback по умолчанию создаёт REQUEST (а не CLARIFICATION) для SLA safety.

**Рабочие часы и 24/7 режим (PR #286):**
13. SLA-таймер корректно приостанавливается вне рабочих часов (09:00-18:00 МСК).
14. В режиме `24/7` SLA-таймер работает круглосуточно.

**Кнопка «Открыть чат» (PR #286):**
15. Кнопка **«Открыть чат»** ведёт по invite-ссылке. Ссылка генерируется автоматически.

**Веб-интерфейс (PR #286):**
16. Страница деталей чата обновляется автоматически (polling каждые 10 сек).

## 4. Обязательная подготовка

### 4.1. Подготовить тестовый чат

Нужен отдельный тестовый чат без реальных клиентов:

1. Тип чата: `group` или `supergroup`.
2. Бот добавлен в чат **с правами администратора** (нужно для invite-ссылки).
3. SLA включён.
4. Порог SLA для удобства теста: **1-2 минуты**.
5. **Режим 24/7 включён** (`is_24x7_mode = true`) — иначе SLA не сработает вечером/в выходные.
6. У чата настроены:
   - минимум один бухгалтер;
   - минимум один менеджер;
   - желательно два бухгалтера для multi-accountant проверки.

> **Важно про рабочие часы:** Если `is_24x7_mode = false`, SLA-таймер считает только рабочее время (Пн-Сб, 09:00-18:00 МСК). Тестирование вечером/ночью/в воскресенье будет выглядеть так, будто система не работает — на самом деле таймер ждёт начала следующего рабочего периода. Для тестирования в любое время включите 24/7.

> **Важно про формат сообщений:** AI-классификатор анализирует текст. Сообщения типа "Тест" или "Сообщение в 16:41" могут быть классифицированы как SPAM. Для создания заявки отправляйте сообщения в формате реального клиентского запроса: **"Когда будет готова декларация?"**, **"Нужна справка 2-НДФЛ"**, **"Не могу оплатить, помогите"**.

### 4.2. Подготовить учётные записи

Перед началом убедиться:

1. Бухгалтер A и бухгалтер B существуют в системе.
2. Менеджер существует в системе.
3. Администратор существует в системе.
4. У всех тестовых внутренних пользователей привязан Telegram через:
   - `/settings/profile`
   - или профиль в веб-интерфейсе.

### 4.3. Подготовить маршруты в веб-панели

Основные экраны:

| Экран | URL |
| --- | --- |
| Чаты | `https://buhbot.aidevteam.ru/chats` |
| Заявки | `https://buhbot.aidevteam.ru/requests` |
| SLA | `https://buhbot.aidevteam.ru/sla` |
| Нарушения | `https://buhbot.aidevteam.ru/violations` |
| Профиль | `https://buhbot.aidevteam.ru/settings/profile` |
| Пользователи | `https://buhbot.aidevteam.ru/settings/users` |

## 5. Базовая проверка настроек ролей

### 5.1. Проверка администратором

1. Войти под администратором.
2. Открыть `/settings/users`.
3. Убедиться, что у тестовых пользователей корректные роли:
   - бухгалтер;
   - менеджер;
   - администратор.
4. Открыть `/chats`.
5. Найти тестовый чат.
6. Проверить настройки чата:
   - ответственный бухгалтер;
   - список бухгалтеров;
   - SLA включён;
   - порог SLA выставлен.

**Ожидаемый результат:**

1. Пользователи с нужными ролями существуют.
2. Тестовый чат виден.
3. Настройки SLA сохранены.
4. Бухгалтеры и менеджеры назначены корректно.

## 6. Сценарий A: создание новой заявки клиентом

### Шаги

1. Зайти в тестовый Telegram-чат под клиентом.
2. Отправить новое сообщение, которое выглядит как запрос клиента.
3. Подождать, пока сообщение обработается системой.
4. Под менеджером или администратором открыть `/requests`.
5. Найти новую заявку.

### Ожидаемый результат

1. Заявка появилась в списке.
2. У заявки корректный текст сообщения.
3. Статус заявки изначально не `closed`.
4. Видно, к какому чату относится запрос.

## 7. Сценарий B: SLA breach и клавиатура у разных ролей

### Подготовка

1. Не отвечать на сообщение клиента до истечения SLA.
2. Дождаться breach-уведомлений.

### Проверка у менеджера

1. Открыть Telegram менеджера.
2. Найти новое SLA alert-сообщение.
3. Проверить кнопки.

**Ожидаемый результат для менеджера:**

1. Alert пришёл.
2. Есть кнопка **«Открыть чат»**.
3. Есть кнопка **«Уведомить бухгалтера»**.
4. Есть кнопка **«Отметить решённым»**.

### Проверка у администратора

Если администратор входит в получателей SLA-alert’ов:

1. Открыть Telegram администратора.
2. Найти то же SLA alert-сообщение.
3. Проверить кнопки.

**Ожидаемый результат для администратора:**

1. Alert пришёл.
2. Есть кнопка **«Открыть чат»**.
3. Есть кнопка **«Уведомить бухгалтера»**.
4. Есть кнопка **«Отметить решённым»**.

### Проверка у бухгалтера

1. Открыть Telegram бухгалтера A.
2. Найти SLA alert-сообщение.
3. Проверить кнопки.

**Ожидаемый результат для бухгалтера:**

1. Alert пришёл.
2. Есть кнопка **«Открыть чат»**.
3. Есть кнопка **«Отметить решённым»**.
4. **Нет** кнопки **«Уведомить бухгалтера»**.

## 8. Сценарий C: менеджер нажимает «Уведомить бухгалтера»

### Шаги

1. В Telegram менеджера открыть breach-alert.
2. Нажать **«Уведомить бухгалтера»**.
3. Проверить реакцию callback.
4. Открыть Telegram бухгалтера A.

### Ожидаемый результат

1. Менеджер получает callback-ответ:
   - **«Уведомление отправлено бухгалтеру»**.
2. Бухгалтер A получает отдельное уведомление.
3. В уведомлении у бухгалтера есть ссылка/кнопка для перехода в чат.
4. Уведомление относится к правильному чату и правильной заявке.

## 9. Сценарий D: уведомление нескольким бухгалтерам

### Подготовка

1. Убедиться, что у тестового чата настроены бухгалтер A и бухгалтер B.
2. Сгенерировать новый breach-alert через новое клиентское сообщение.

### Шаги

1. Под менеджером нажать **«Уведомить бухгалтера»**.
2. Проверить Telegram бухгалтера A.
3. Проверить Telegram бухгалтера B.

### Ожидаемый результат

1. Бухгалтер A получил уведомление.
2. Бухгалтер B получил уведомление.
3. Уведомления не дублируются одному и тому же пользователю.
4. Менеджер видит успешный callback.

## 10. Сценарий E: fallback legacy-конфигурации по username

> Этот сценарий нужен только если есть подготовленный legacy-чат, где у бухгалтеров есть `accountantUsernames`, но нет `accountantTelegramIds`.

### Шаги

1. Открыть заранее подготовленный legacy-чат.
2. Создать новую заявку.
3. Дождаться breach-alert.
4. Нажать **«Уведомить бухгалтера»**.
5. Если DM не доставлен, проверить исходный групповой чат.

### Ожидаемый результат

1. Система не падает.
2. Если известны несколько username бухгалтеров, в fallback-сообщении упоминаются **все**, а не только один.
3. Callback не возвращает ложный успех без фактической доставки.

## 11. Сценарий F: ручной resolve менеджером

### Шаги

1. Создать новую заявку от клиента.
2. Дождаться breach-alert у менеджера.
3. Под менеджером нажать **«Отметить решённым»**.
4. Проверить callback в Telegram.
5. Открыть `/requests` под менеджером или администратором.
6. Найти эту заявку.

### Ожидаемый результат

1. Менеджер видит callback:
   - **«Отмечено как решённое»**.
2. Исходное Telegram alert-сообщение меняется на resolved-вид.
3. В веб-панели заявка уходит в состояние `closed`.
4. Заявка **не** становится `answered`, если бухгалтер реально не отвечал.
5. У заявки не появляется искусственное время ответа от ручного resolve.

## 12. Сценарий G: stale resolve после ручного закрытия

### Подготовка

Нужно, чтобы один и тот же breach-alert пришёл минимум двум получателям:

1. менеджеру;
2. администратору или второму менеджеру.

### Шаги

1. Под менеджером A нажать **«Отметить решённым»**.
2. Не открывая новое сообщение, перейти к старой копии alert’а у менеджера B или администратора.
3. Нажать старую кнопку **«Отметить решённым»**.

### Ожидаемый результат

1. Второй пользователь получает callback:
   - **«Запрос уже закрыт»**.
2. Система не падает.
3. Состояние заявки не меняется повторно.

---

## 13. Сценарий H: stale resolve после реального ответа бухгалтера

### Шаги

1. Создать новую заявку от клиента.
2. Дождаться breach-alert у менеджера.
3. Не нажимать `resolve`.
4. Под бухгалтером ответить клиенту в самом Telegram-чате как на реальную заявку.
5. Убедиться, что заявка обработалась как настоящий ответ бухгалтера.
6. Вернуться к старой копии breach-alert у менеджера.
7. Нажать **«Отметить решённым»**.

### Ожидаемый результат

1. Заявка остаётся с результатом реального ответа бухгалтера.
2. Менеджер получает callback:
   - **«Запрос уже отмечен как решённый»**.
3. Ручной callback не должен перетирать реальный ответ.

## 14. Сценарий I: ручной resolve не должен портить статистику бухгалтера

### Шаги

1. Под бухгалтером открыть Telegram-бота.
2. Выполнить команду `/mystats`.
3. Зафиксировать текущее значение:
   - отвечено;
   - среднее время ответа;
   - SLA compliance.
4. Создать новую клиентскую заявку.
5. Дождаться breach-alert.
6. Под менеджером закрыть заявку через **«Отметить решённым»**.
7. Снова под бухгалтером выполнить `/mystats`.

### Ожидаемый результат

1. Ручной `resolve` не должен увеличивать счётчик реально отвеченных заявок.
2. Среднее время ответа не должно меняться из-за ручного `resolve`.
3. Статистика бухгалтера не должна выглядеть так, как будто он реально ответил клиенту.

## 15. Сценарий J: отображение статуса в веб-интерфейсе

### Шаги

1. Повторить два кейса:
   - ручной `resolve`;
   - реальный ответ бухгалтера.
2. После каждого кейса открыть:
   - `/requests`
   - `/requests/{id}`
   - `/violations`
   - `/sla`

### Что проверить

#### После ручного `resolve`

1. Заявка не выглядит как реально отвеченная бухгалтером.
2. Статус закрыт корректно.
3. Нет странного «времени ответа», если ответа не было.

#### После реального ответа бухгалтера

1. Заявка отображается как реально отвеченная.
2. Время ответа рассчитывается корректно.
3. Старые alert-кнопки больше не могут сломать статус.

## 16. Сценарий K: права ролей

### Бухгалтер

Проверить:

1. бухгалтер получает свои alert’ы;
2. бухгалтер не видит кнопку **«Уведомить бухгалтера»**;
3. бухгалтер может открыть чат;
4. бухгалтер не ломает callback-path.

### Менеджер

Проверить:

1. менеджер получает alert;
2. менеджер видит обе action-кнопки;
3. менеджер может уведомить бухгалтера;
4. менеджер может закрыть заявку вручную.

### Администратор

Проверить:

1. администратор может проверить настройки пользователей и чатов;
2. если администратор входит в получателей alert’ов, он видит manager-like клавиатуру;
3. stale callback у администратора не ломает состояние заявки.

### Клиент

Проверить:

1. клиент создаёт запрос обычным сообщением;
2. клиентское сообщение попадает в систему как заявка;
3. дальнейшие действия внутренних ролей не теряют связь с исходной заявкой.

## 17. Сценарий L: классификация сообщений

### Подготовка

1. В тестовом чате должен быть включён SLA.
2. Отправлять сообщения от клиента (НЕ от бухгалтера).

### L.1 Сообщение-запрос (ожидаем REQUEST)

Отправить: **"Когда будет готова декларация?"** или **"Нужна справка 2-НДФЛ"**.

**Ожидаемый результат:**
1. Создана заявка (ClientRequest) со статусом `pending`.
2. `classification = REQUEST`, `classification_model = openrouter`.
3. SLA-таймер запущен (`sla_timer_started_at` не null).

### L.2 Сообщение-спам (ожидаем SPAM)

Отправить: **"Ок"** или **"Понял, жду"** или **"👍"**.

**Ожидаемый результат:**
1. Заявка НЕ создаётся.
2. Сообщение логируется в `chat_messages`, но в `client_requests` записи нет.

### L.3 Благодарность (ожидаем GRATITUDE)

Отправить: **"Спасибо большое за помощь!"**.

**Ожидаемый результат:**
1. Заявка НЕ создаётся.
2. Классификация GRATITUDE.

### L.4 Неоднозначное сообщение (safety default)

Отправить сообщение без явных маркеров запроса: **"Информация по квартальному отчёту"**.

**Ожидаемый результат:**
1. AI должен классифицировать. Если AI вернёт REQUEST — создаётся заявка с SLA.
2. Если AI недоступен, keyword fallback по умолчанию создаст REQUEST (SLA safety).

## 18. Сценарий M: рабочие часы и 24/7 режим

### M.1 Тест вне рабочих часов (режим НЕ 24/7)

**Подготовка:** Убедиться, что у чата `is_24x7_mode = false`. Тестировать вечером после 18:00 МСК или в воскресенье.

1. Отправить сообщение-запрос от клиента.
2. Подождать больше SLA-порога.

**Ожидаемый результат:**
1. Заявка создана, SLA-таймер запущен.
2. Алерт **НЕ приходит** сразу — таймер ждёт следующего рабочего периода.
3. Алерт придёт в начале рабочего времени (например, понедельник 09:01 МСК).

### M.2 Тест в режиме 24/7

**Подготовка:** Установить `is_24x7_mode = true` для тестового чата (через настройки или БД).

1. Отправить сообщение-запрос вечером/ночью/в выходной.
2. Подождать SLA-порог.

**Ожидаемый результат:**
1. Алерт приходит через указанный порог, **независимо от времени суток и дня недели**.

### M.3 Тест в рабочие часы

**Подготовка:** Тестировать в будний день с 09:00 до 18:00 МСК.

1. Отправить сообщение-запрос.
2. Подождать SLA-порог.

**Ожидаемый результат:**
1. Алерт приходит точно через заданный порог SLA.

## 19. Сценарий N: кнопка «Открыть чат» и invite-ссылка

### N.1 Кнопка «Открыть чат» у свежего алерта

1. Дождаться нового breach-alert.
2. Нажать кнопку **«Открыть чат»**.

**Ожидаемый результат:**
1. Telegram открывает групповой чат.
2. Ссылка ведёт в правильный чат (не в другой).

### N.2 Если invite-ссылка отсутствовала

При первом алерте для чата без invite-ссылки система автоматически генерирует её через Telegram API.

**Ожидаемый результат:**
1. Кнопка **«Открыть чат»** появляется в алерте.
2. После первого алерта ссылка сохранена в БД (последующие алерты используют её).

> **Важно:** Бот должен быть администратором чата с правом **Invite Users**. Если прав нет, вместо invite-ссылки используется `t.me/c/` fallback (работает только для участников группы).

## 20. Сценарий O: веб-интерфейс обновляется автоматически

### O.1 Обновление после resolve через Telegram

1. Открыть страницу чата в веб-интерфейсе: `/chats/{id}`.
2. В Telegram нажать **«Отметить решённым»** на алерте для заявки из этого чата.
3. **Не обновлять страницу вручную.**

**Ожидаемый результат:**
1. В течение 10 секунд статус заявки на странице обновляется автоматически.
2. Не нужно нажимать F5 или перезагружать страницу.

### O.2 Обновление после ответа бухгалтера

1. Открыть страницу чата в веб-интерфейсе.
2. В Telegram бухгалтер отвечает на сообщение клиента.
3. **Не обновлять страницу вручную.**

**Ожидаемый результат:**
1. Новое сообщение появляется в ленте (polling каждые 10 сек).
2. Статус заявки обновляется до `answered`.

## 21. Сценарий P: Telegram-алерт при сбое AI-классификатора

> Этот сценарий проверяется при естественных сбоях OpenRouter или через намеренную смену модели на несуществующую.

### Шаги

1. При следующем сбое AI (если OpenRouter вернёт ошибку) проверить Telegram менеджера.

**Ожидаемый результат:**
1. Менеджер получает сообщение с текстом: **«BuhBot: AI-классификатор недоступен»**.
2. Сообщение содержит название модели и текст ошибки.
3. Алерт приходит **не чаще раза в 5 минут** (rate limiting).
4. При этом SLA всё равно работает — keyword fallback создаёт REQUEST.

## 22. Быстрый smoke-маршрут для короткой проверки

Если времени мало, обязательно пройти минимум:

1. Клиент отправляет **"Когда будет готова декларация?"** — создаётся заявка, SLA запущен.
2. Клиент отправляет **"Ок"** — заявка НЕ создаётся (SPAM).
3. Менеджер получает breach-alert с 3 кнопками.
4. Бухгалтер получает breach-alert **без** кнопки «Уведомить бухгалтера».
5. Менеджер нажимает **«Уведомить бухгалтера»** — бухгалтер получает уведомление.
6. Менеджер нажимает **«Открыть чат»** — Telegram открывает правильный чат.
7. Менеджер нажимает **«Отметить решённым»** — заявка `closed` (не `answered`).
8. Вторая копия alert’а даёт **«Запрос уже закрыт»**.
9. Веб-интерфейс обновляется без ручного F5.
10. Реальный ответ бухгалтера не перетирается старым `resolve`.

## 24. Итоговый критерий успеха

Система считается работоспособной, если:

**SLA и callback:**
1. notify-path работает у менеджера и администратора;
2. бухгалтер не видит лишнюю кнопку;
3. multi-accountant доставка работает;
4. manual resolve закрывает заявку, но не подменяет реальный ответ;
5. stale кнопки не перетирают актуальное состояние;
6. статистика бухгалтера не искажается после ручного закрытия;

**Классификация:**
7. AI классифицирует сообщения-запросы как REQUEST;
8. SPAM/GRATITUDE не создают заявки;
9. При сбое AI — fallback на вторую модель, затем keyword (default REQUEST);
10. Менеджеры получают Telegram-алерт при полном отказе AI;

**Рабочие часы:**
11. SLA-таймер учитывает рабочие часы (09:00-18:00 МСК, Пн-Сб);
12. Режим 24/7 корректно отключает учёт рабочих часов;

**Веб-интерфейс:**
13. Кнопка «Открыть чат» работает (invite-ссылка генерируется автоматически);
14. Страница деталей чата обновляется автоматически (10 сек);
15. Состояние заявки в вебе согласовано с Telegram-callback.
