## Ответ на критические замечания

### «Бот не подтверждает клиенту получение сообщения»

Автоответ на текстовые сообщения не входил в реализованный скоуп. Для файлов подтверждение есть и работает («Файл получен!»). Если нужен автоответ и на текст — сделаем, это новая задача, а не баг.

### «Бухгалтер не получает уведомление о нарушении SLA»

Код уведомлений рабочий: L1 → личка бухгалтеру, L2+ → менеджерам. Для срабатывания нужно три условия: `slaEnabled = true`, Telegram ID бухгалтера привязан, мониторинг включён. Если после обновления настройка слетела — тишина, и в интерфейсе это не видно. Реальный пробел — нет диагностики в UI. Команда `/diagnose` в боте есть, но в вебе аналога нет.

### «Нарушения не фиксируются в интерфейсе»

Следствие предыдущего пункта. Если таймер не запустился (SLA выключен, ID не привязан) — нарушения не создаются, показывать нечего. Логика фиксации рабочая, но она downstream от конфигурации.

### «internalChatId — без него уведомления не уходят»

Ошибка в анализе. `internalChatId` — дополнительный канал для логирования в общий чат команды. Основные уведомления бухгалтеру и менеджерам идут через `queueAlert()` и не зависят от этой настройки.

### «Уведомления улетели в клиентский чат» (26 декабря)

Был реальный баг. Исправлен.

### «dev-версия на проде» (19 февраля)

Инфраструктурная ошибка при деплое. Не проблема кода, но проблема процесса.

### «Невозможно массово добавить бухгалтеров»

Справедливо. Сейчас: каждому написать боту → получить ID → вручную ввести цифры. Для 25 человек неприемлемо. Нужен механизм группового онбординга.

### «Документации нет»

Справедливо. Пошаговой инструкции нет.

### «Цикл починили → сломалось»

Справедливое наблюдение. Причина: нет smoke-тестов после деплоя и совместной верификации.
