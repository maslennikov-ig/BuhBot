### F1 — Worker and recovery tests do not assert `slaBreachedAt` is written (P1)

**PR:** #301
**File(s):** `backend/src/queues/__tests__/sla-timer.worker.test.ts`, `backend/src/services/sla/__tests__/timer.service.test.ts`

**Observation:**
The primary regression protection for `slaBreachedAt` is the schema-contracts test (`schema-contracts.test.ts`) which confirms the tRPC output schema accepts the field. However, neither `sla-timer.worker.test.ts` nor `timer.service.test.ts` (the `recoverPendingSlaTimers` test) assert that `clientRequest.update` is called with `slaBreachedAt: expect.any(Date)`. The worker test only checks `bot.telegram.sendMessage`; the recovery test only checks `queueAlert`.

**Why it matters:**
If someone later removes or misnames the `slaBreachedAt` field from the `data:` block in either setter, all existing tests will continue to pass. The tRPC schema only proves the field can be accepted by the output — not that it is actually populated by the write paths.

**Suggested fix:**
In `sla-timer.worker.test.ts`, add:
```typescript
expect(mockPrisma.$transaction).toHaveBeenCalled();
// Or capture the tx mock and assert:
expect(mockClientRequest.update).toHaveBeenCalledWith(
  expect.objectContaining({
    data: expect.objectContaining({
      slaBreached: true,
      slaBreachedAt: expect.any(Date),
    }),
  })
);
```

In `timer.service.test.ts` (recovery test), add:
```typescript
expect(mockPrisma.clientRequest.update).toHaveBeenCalledWith(
  expect.objectContaining({
    data: expect.objectContaining({
      slaBreached: true,
      slaBreachedAt: expect.any(Date),
    }),
  })
);
```

**Test recommendation:** See above — add assertions to existing tests, no new test file needed.

```tsx
  ...

```

### F4 — `isSlaExcessSevere` zero-SLA floor is an undocumented business decision (P2)

**PR:** #301
**File(s):** `frontend/src/lib/format-duration.ts:87`

**Observation:**
```typescript
if (slaMinutes <= 0) return excessMinutes >= MINUTES_PER_HOUR; // 60 min floor
```
When `slaMinutes` is 0 or negative (which the schema allows as nullable and the page coerces with `|| 60`), the function silently uses a 1-hour severity threshold. However, the page-level code already handles `slaMinutes = 0` by substituting 60: `const slaMinutes = req.slaWorkingMinutes || 60`. So `isSlaExcessSevere` should never actually receive `slaMinutes <= 0` from the violations page. The defensive branch is unreachable in production. The test for it passes but it tests dead code in the current call site.

**Why it matters:**
The defensive floor is reasonable, but it silently masks a data integrity issue (why would `slaMinutes` be 0 in the DB?). If someone calls the utility from a different call site that doesn't apply the `|| 60` guard, the behavior will be surprising (2 × 0 = 0, so excessSevere would be true for any positive excess — but the floor overrides this to 60 min). A comment explaining the floor's rationale and that it is defensive-only would prevent misuse.

**Suggested fix:**
```typescript
// Defensive: slaMinutes should never reach 0 in production because
// the caller normalises with `|| 60`, but guard anyway. Use 1 h as
// a sensible minimum threshold matching the global SLA default.
if (slaMinutes <= 0) return excessMinutes >= MINUTES_PER_HOUR;
```

### F5 — Ticking `useMemo` recomputes all rows every 30 s including resolved breaches (P2)

**PR:** #301
**File(s):** `frontend/src/app/violations/page.tsx:221`

**Observation:**
`sortableViolations` has `[violationsQuery.data, nowTick]` as its dependency array. Every 30 seconds `nowTick` changes, triggering a full re-map of all violations — including ones where `isOpenBreach = false` (response recorded). For those rows `computeSlaExcessMinutes` immediately returns `(responseAt - receivedAt) - slaMinutes`, which is stable and does not depend on `now`. The extra work is pure overhead.

**Why it matters:**
For typical usage (tens of violations) this is negligible. But a customer with hundreds of historical violations in the 30-day window will cause unnecessary React reconciliation every 30 s. More importantly, re-running the map causes all row components to receive new object references, potentially defeating `React.memo` on child components.

**Suggested fix (optional, can defer):**
Split into two memos: one stable memo for resolved rows (no `nowTick` dep), one ticking memo for open breaches only. Or simply ensure row rendering components are stable via `key` props and accept the current approach as intentionally simple.

### F6 — Live-ticking excess counter lacks ARIA `aria-live` region (P2)

**PR:** #301
**File(s):** `frontend/src/app/violations/page.tsx:443–462`

**Observation:**
The excess cell (`+{formatDuration(violation.excessMinutes)}`) updates every 30 seconds for open breaches. There is no `aria-live` region around the counter, no `role="timer"`, and no `aria-label` distinguishing open vs closed breaches for screen reader users (the distinguishing dot `•` is `aria-hidden`).

**Why it matters:**
For screen reader users, the ticking value is invisible. The tooltip (title attribute) does explain the open/closed state, but `title` is not reliably announced by screen readers. This is an a11y regression relative to the non-ticking version.

**Suggested fix:**
```tsx
<span
  role="timer"
  aria-label={
    violation.isOpenBreach
      ? `Превышение SLA: +${formatDuration(violation.excessMinutes)}, ответа нет`
      : `Превышение SLA: +${formatDuration(violation.excessMinutes)}`
  }
  className={...}
>
```
Or wrap the counter text in an `aria-live="polite"` region (noting that polite announcements every 30 s may be annoying; `aria-live="off"` with a descriptive `aria-label` is likely the better choice).

### F8 — Shared `popover.tsx` base z-50 creates a systemic fragility (nit)

**PR:** #300 (systemic)
**File(s):** `frontend/src/components/ui/popover.tsx:22`

**Observation:**
The shared `PopoverContent` in `popover.tsx` defaults to `z-50`. Any future component that places a Popover inside a `z-[1200]` modal will silently render behind the modal overlay. The fix in this PR adds `z-[1300]` per-callsite, but this is a pattern that must be remembered by every future developer.

**Why it matters:**
This is a systemic concern rather than a bug in this PR. The root fix would be to change the shared default to a higher value, but that risks conflicting with other Radix portals (Select, DropdownMenu, etc.) that might have their own z-stacking. Alternatively, a `WithModalZIndex` wrapper or a design token for modal-relative z-indexes would solve this structurally.

**Suggested fix (longer term):**
Document the z-index stacking contract in a comment in `popover.tsx`:
```tsx
// z-50 is the default for non-modal contexts. Inside a z-[1200] modal,
// callers must pass z-[1300] or higher via className. See AccountantSelect
// for reference (gh-289).
```

- **PR #301 (gh-290):** The formula `x = responseAt || slaBreachedAt || now` is semantically correct: it produces the most accurate fixed-point for resolved breaches and a live-updating value for unresolved ones. The `toMs()` helper in `format-duration.ts` transparently handles both `Date` and ISO string inputs, which is essential given tRPC without SuperJSON serializes `Date` → ISO string in JSON (no transformer configured). The adaptive unit formatter (`м/ч м/д ч/н д`) is a significant UX improvement for long-running breaches. The backfill migration's three-priority COALESCE strategy is sound: `responseAt` (most accurate) → synthesized breach time → 60-min legacy fallback. The migration correctly guards with `sla_breached_at IS NULL` to prevent double-writes. Recovery path safety analysis: the `recoverPendingSlaTimers` query filters by `status: { in: ['pending'] }`, and the worker sets `status: 'escalated'` atomically in the same transaction as `slaBreachedAt`. So the recovery path and the live worker operate on disjoint sets of rows, eliminating the stomp risk. `slaAttempts: 1` in queue config means no retry → no re-entry risk for the breach job.

- **Backfill concurrent write safety:** The `ADD COLUMN` is `NULLABLE` so existing rows remain valid immediately. The `UPDATE` backfill runs after the column exists and is conditional (`sla_breached_at IS NULL`). On Supabase PostgreSQL 15, `ADD COLUMN ... TIMESTAMPTZ` with nullable default is a metadata-only operation (no table rewrite), so it completes instantly and does not hold an `ACCESS EXCLUSIVE` lock for the full backfill duration. The `UPDATE` takes row-level locks only on breached rows. Risk is low.

- **Test coverage for format-duration:** 15 tests covering null/NaN/negative/zero, all unit tiers, fractional truncation, ISO string inputs, and the `isSlaExcessSevere` 2× threshold. The test for `isSlaExcessSevere` with `slaMinutes=0` correctly exercises the defensive floor.

- **`make_interval(mins => ...)` on PostgreSQL 15:** Valid syntax. `make_interval` with named parameter `mins` is supported since PostgreSQL 10.
