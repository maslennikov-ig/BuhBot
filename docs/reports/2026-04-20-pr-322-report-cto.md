Сергей, привет. Отработал по последним комментариям в issues #313 и #294 с прода.

**Что делал**

1. `#313` chat picker выдавал 400 — frontend в code-review M1 поднял limit до 500, а backend Zod остался на max(100). Поднял backend тоже до 500 + тест на границу.

2. `#294` на детали опроса все доставки показывались как красные «Ошибка». Копнул прод-БД: все 8 `failed`-строк имели `skip_reason LIKE 'cooldown:%'` и `error_message = null` — это cooldown-скипы gh-292, а не настоящие падения Telegram. Worker писал `status='failed'` и UI красил красным.
   Исправил: новый enum `DeliveryStatus.skipped`, две отдельные миграции (ALTER TYPE + backfill — по правилам PostgreSQL), worker пишет `skipped` на cooldown, `failed` оставил только для реальных ошибок sendMessage. UI расширил: серый бейдж «Пропущен» + фильтр + tooltip со `skipReason`. Фронт имеет fallback для неизвестных статусов на случай раскатки.

3. В процессе тестов нашёл побочный прод-блокер — quarter-пресет `2026-Q2` падал с `400 Range span 91.0d exceeds maximum 90d`. Баг давний, из gh-292: `surveyMaxRangeDays=90` дефолт, а реальные кварталы 90–92 дня.
   Исправил: добавил опциональный `bypassMaxRangeCheck` во внутренний API `createCampaign`, роутер включает флаг ровно для quarter-режима. Abuse-гуард на custom-range остался как был. Тесты покрывают оба направления.
