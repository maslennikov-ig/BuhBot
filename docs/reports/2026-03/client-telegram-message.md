Ответ на аналитику — по каждому пункту критики

Спасибо за детальный разбор. Большая часть наблюдений справедлива, по нескольким моментам нужны уточнения.

1. «Бот не подтверждает клиенту получение сообщения»
Автоответ на текстовые сообщения не входил в реализованный скоуп. Для файлов подтверждение есть и работает. Если нужен автоответ и на текст — сделаем, это новая задача.

2. «Бухгалтер не получает уведомление»
Код уведомлений рабочий: при нарушении SLA алерт уходит бухгалтеру в личку, при эскалации — менеджерам. Но для этого нужно три условия: SLA включён на чате, Telegram ID бухгалтера привязан, мониторинг активен. Если после обновления что-то слетело — система молчит, и в интерфейсе это не видно. Это реальный пробел — признаём, будем исправлять.

3. «Нарушения не видны в интерфейсе»
Следствие пункта 2. Если таймер не запустился из-за конфигурации — нарушения не создаются, показывать нечего. Сама логика фиксации рабочая.

4. «internalChatId — без него уведомления не уходят»
Здесь в анализе неточность. internalChatId — это дополнительный канал для логирования в общий чат команды. Основные уведомления бухгалтеру и менеджерам идут через отдельный механизм и не зависят от этой настройки.

5. «Уведомления улетели в клиентский чат»
Был реальный баг в декабре. Исправлен.

6. «dev-версия на проде»
Ошибка при деплое, не проблема кода. Закрываем автотестами после деплоя.

7. «Невозможно массово добавить бухгалтеров»
Справедливо. Для 25 человек текущий механизм не подходит. Нужна доработка — отдельная задача.

8. «Документации нет»
Справедливо. Пошаговой инструкции нет, это приоритет.

9. «Цикл починили → сломалось»
Справедливое наблюдение. Причина — нет автотестов после деплоя и совместной проверки. Решаем.
