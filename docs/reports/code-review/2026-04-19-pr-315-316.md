**[backend/src/bot/handlers/survey.handler.ts:186ŌĆō191]** Timer accumulation on
repeated votes: `setTimeout` handle is never stored or cancelled, so each vote
change by the same user registers an additional 10-minute timer against the same
Map key. When the first timer fires, it deletes the entry even though the user
may have voted again 30 seconds later and the entry was refreshed.

Concrete scenario: User votes at t=0 (timer A scheduled), changes vote at t=1m
(timer B scheduled, entry updated). At t=10m timer A fires and deletes the
entry, which now holds the updated `voteId` from the second vote. Timer B fires
at t=11m on an already-deleted key (harmless no-op), but between t=10m and
t=11m the comment window is incorrectly closed.

This bug exists on `main` as well (pre-existing), but the PR adds a new code
path (static keyboard, no `editMessageText` guard) that makes repeated voting
more likely, so it is worth flagging here.

Recommended fix:

```typescript
// Store timer handle per key so it can be cancelled on re-vote.
const commentTimers = new Map<string, ReturnType<typeof setTimeout>>();

// In the rating handler, before awaitingComment.set():
const existingTimer = commentTimers.get(key);
if (existingTimer) clearTimeout(existingTimer);

const timer = setTimeout(() => {
  awaitingComment.delete(key);
  commentTimers.delete(key);
}, 10 * 60 * 1000);
commentTimers.set(key, timer);
```

Apply the same `clearTimeout` in the remove handler when deleting the entry.

**[backend/src/bot/handlers/survey.handler.ts:167 & 257]** The error path in
the `survey:rating` handler calls `editMessageText(SURVEY_EXPIRED_MESSAGE)` to
replace the message text when the survey is closed or not found (lines 132, 205,
210). This is correct behaviour. However, the same edit will now overwrite the
static keyboard that was added in this PR (`createSurveyRatingKeyboard` returns
both text and markup). `editMessageText` with only `parse_mode` does not preserve
the existing reply markup ŌĆö Telegram will detach the keyboard. After the message
is edited to `SURVEY_EXPIRED_MESSAGE` the buttons become inert because they have
no keyboard frame; that is the desired end state. Verify that this is the
intended UX when a survey expires mid-session (buttons disappear for all users),
and if so, add a one-line comment clarifying the intentional keyboard removal in
the error branch so future reviewers do not treat it as a missing call to
`editMessageReplyMarkup`.

**[backend/src/bot/handlers/__tests__/survey.handler.test.ts & multiuser.test.ts]**
There is no test for the case where `answerCbQuery` itself throws (Telegram
returns `400 Bad Request: query is too old` for callbacks older than ~48 s after
the button was pressed). The success path awaits `answerCbQuery` without a
try-catch, so an expired callback query would propagate the error to the outer
catch block and trigger the generic `'Error processing rating'` reply ŌĆö which
itself calls `answerCbQuery` again and would throw a second time, leaving the
outer catch's `await ctx.answerCbQuery('Error processing rating')` unhandled.
This is pre-existing behaviour (not introduced by this PR) but the PR is a good
opportunity to add the guard.

Recommended fix: wrap `answerCbQuery` in a best-effort helper or add
`.catch(() => {})` for the success-path toast, since Telegram's answer deadline
is independent of our business logic.

### Summary

**Verdict: approve-with-changes.**

The core fix is correct and well-motivated. Removing `editMessageReplyMarkup`
and `editMessageText` from the success path is the right solution to the
multi-user keyboard regression. The regression guard tests are clear and will
catch future regressions reliably.

Two High-priority items should be addressed before merge: the misleading
comment-collection hint in the toast when `chatId` is missing, and the
accumulated `setTimeout` handles that can silently close the comment window
after a vote change. Both issues are pre-existing on `main`, but this PR makes
repeated voting more likely and is therefore the natural place to fix them.

Test count sanity: the PR adds 2 new `it()` blocks in `multiuser.test.ts` and
rewrites 2 existing ones in `survey.handler.test.ts`. The total number of
survey-related test cases (5 in multiuser + 3 in unit) is reasonable for the
scope of the fix, though the error path in `survey:remove` (closed survey) is
only covered implicitly via the older shared test; a dedicated case for
`SurveyClosedError` in the remove handler would strengthen coverage.

are used throughout the workflow (pre-existing, not introduced by this PR).
The latest stable major for both actions is `v4`. Version 6 does not exist as a
published release for either action; GitHub Actions resolves `@v6` by walking
tags and, if no `v6` tag exists, may resolve to `main` or fail at runtime
depending on the repository's tag layout. This is a latent risk in the existing
CI and is unrelated to the docker action bumps in this PR, but since the PR
touches the same file it is a good time to fix it.

Verify with `gh release list -R actions/checkout` and
`gh release list -R actions/setup-node` and pin to `@v4` (current LTS).

**[.github/workflows/ci.yml:174]** The `build-docker` job (and `test` job at
line 292) references `needs.detect-release-pr.outputs.should_skip_heavy` in its
`if:` condition but does not declare `needs: [detect-release-pr]`. Per GitHub
Actions semantics, an `if:` expression that references `needs.X` without `X` in
`needs:` evaluates the expression against an empty context, which means the
condition may behave unexpectedly (the output resolves to empty string, which
differs from `'true'`). In practice the job appears to run because the
inequality `!= 'true'` is satisfied by an empty string, but this is relying on
accidental correctness. This is pre-existing and not introduced by this PR, but
again the file is being touched.

Recommended fix:

```yaml
build-docker:
  name: Build Docker Images
  needs: [detect-release-pr]   # add this line
  runs-on: ubuntu-latest
```
