## PR #319 — fix(sla): expose chat SLA threshold (`fix/gh-290-old-violations-zero-excess`)

### Critical

None.

### High

**[`backend/prisma/migrations/20260419000000_backfill_sla_breached_at/migration.sql` — no transaction wrapper]**
The backfill migration contains two bare `UPDATE` statements with no `BEGIN`/`COMMIT` block. If the second statement (orphan-chat fallback) fails mid-run — e.g. due to a lock timeout on a large table — the first UPDATE has already committed and the migration is left in a half-applied state. Prisma does not automatically wrap DML-only migration files in a transaction.

Fix: wrap both statements in a single explicit transaction so they are atomic.

```sql
BEGIN;

UPDATE "public"."client_requests" cr
SET "sla_breached" = true,
    "sla_breached_at" = cr."received_at" + make_interval(mins => COALESCE(c."sla_threshold_minutes", 60))
FROM "public"."chats" c
WHERE cr."chat_id" = c."id"
  AND cr."sla_breached_at" IS NULL
  AND ( ... );

UPDATE "public"."client_requests" cr
SET "sla_breached" = true,
    "sla_breached_at" = cr."received_at" + make_interval(mins => 60)
WHERE cr."sla_breached_at" IS NULL
  AND NOT EXISTS (SELECT 1 FROM "public"."chats" c WHERE c."id" = cr."chat_id")
  AND ( ... );

COMMIT;
```

**[`backend/src/api/trpc/routers/sla.ts:242` — `chat?.slaThresholdMinutes ?? 60` optional chaining masks a latent data integrity gap]**
`formatRequestOutput` receives `chat: Chat | null`. For the violations list (`getRequests`), every `clientRequest.findMany` call already includes `chat: true`. However, `chat` can be `null` if the chat row was hard-deleted after the request was created. In that case, `formatRequestOutput` silently falls back to `60` instead of the actual threshold for that chat, so the displayed excess will be computed against the wrong threshold.

This is not a regression introduced by this PR — the same fallback existed before for `chatTitle`. But it is now a new silent correctness risk specifically for the excess calculation (the value is now load-bearing, whereas `chatTitle` is decorative).

Fix: add a warning-level log when `chat` is null and a breach exists, so the condition is observable in production.

```typescript
if (!chat && request.slaBreached) {
  logger.warn('formatRequestOutput: chat is null for breached request, threshold defaulting to 60', {
    requestId: request.id,
    chatId: String(request.chatId),
    service: 'sla-router',
  });
}
slaThresholdMinutes: chat?.slaThresholdMinutes ?? 60,
```

### Medium

**[`frontend/src/lib/__tests__/format-duration.test.ts:108` — misleading comment, correct assertion]**
The inline comment on the legacy unanswered test reads `// 2 days ago, SLA 60 → excess 60+24*60-60 = 2820`, but `60+24*60-60 = 1440` (one day), not 2820. The expected assertion value `2 * 24 * 60 - 60 = 2820` is mathematically correct — the comment formula is just wrong. A future reader could incorrectly "fix" the comment and then mistrust the assertion.

Fix: update the comment to match the assertion.

```typescript

```

**[`backend/src/api/trpc/routers/sla.ts` — no schema-contract test for `slaThresholdMinutes` in `RequestOutput`]**
The PR adds `slaThresholdMinutes: z.number()` (non-nullable) to the tRPC output schema. There is no test that asserts the field is present and correctly valued in the serialised output of `getRequests` or `getRequest`. If a future refactor accidentally changes this field to `z.number().nullable()` or removes it, the regression would be silently invisible until the violations page displays wrong data again.

Recommendation: add at minimum one integration-style test (mock Prisma, call `formatRequestOutput`, assert `slaThresholdMinutes` equals `chat.slaThresholdMinutes`). A snapshot test on the zod schema shape is also acceptable.

**[`backend/prisma/migrations/20260419000000_backfill_sla_breached_at/migration.sql` — unanswered-breach fallback uses `now()` at migration time]**
The second `UPDATE` (orphan-chat fallback) includes a condition `AND now() > cr."received_at" + make_interval(mins => 60)`. This is correct in intent but `now()` is evaluated at statement execution time, not at a stable reference point. If the migration runs exactly at midnight and the DB clock drifts, a request that breached 0.1 seconds ago could be missed. More importantly, on a large dataset this `now()` call is resolved once per row evaluation, which is fine in PostgreSQL, but is worth documenting.

This is a low-severity correctness note, not a blocking defect. The unanswered-breach case is already defensively handled in the frontend (fall-through to live `now` tick). No immediate action required, but the comment in the migration should be updated to document the `now()` is evaluated at run time.

**[`backend/prisma/migrations/20260419000000_backfill_sla_breached_at/migration.sql` — `sla_breached = true` set unconditionally for orphans]**
The fallback UPDATE sets `sla_breached = true` alongside `sla_breached_at`. On rows where `sla_breached` was already `false` (i.e., the live worker decided the request was not breached even with elapsed time > 60), this UPDATE overrides that decision for any chat-less orphan row. The comment says "we cannot join chats reliably" — true, but the precedence is worth documenting. If a chat is soft-deleted (has a `deleted_at` timestamp) rather than hard-deleted, the `NOT EXISTS` clause will still catch it and apply the 60-minute default, potentially overriding a per-chat threshold that was different.

Verify: are there soft-deleted chats whose `sla_threshold_minutes` was not 60? If so, consider changing `NOT EXISTS (SELECT 1 FROM "public"."chats" c WHERE c."id" = cr."chat_id")` to `NOT EXISTS (SELECT 1 FROM "public"."chats" c WHERE c."id" = cr."chat_id" AND c."deleted_at" IS NULL)` (separate pass for soft-deleted) or joining soft-deleted chats too.

### Low

**[`frontend/src/lib/format-duration.ts:89` — JSDoc caller contract buried, not enforced]**
The JSDoc for `computeSlaExcessMinutes` documents that `slaMinutes` MUST be the configured threshold, not `slaWorkingMinutes`. This is a prose contract only — the TypeScript type is `number`. There is no runtime guard or type-branded enforcement. Given that this confusion was the root cause of the original bug, consider renaming the parameter to `slaThresholdMinutes` to make the intent self-documenting.

```typescript
export function computeSlaExcessMinutes(params: {
  receivedAt: Date | string | null | undefined;
  responseAt: Date | string | null | undefined;
  slaBreachedAt: Date | string | null | undefined;
  slaThresholdMinutes: number;  // renamed from slaMinutes
  now: Date;
})
```

**[`frontend/src/app/violations/page.tsx:206` — `req.slaBreachedAt` date coercion redundant]**
`slaBreachedAt: req.slaBreachedAt ? new Date(req.slaBreachedAt) : null` — tRPC with superjson off coerces Date fields to ISO strings on the wire, so `new Date(...)` is correct. However, `computeSlaExcessMinutes` already handles `Date | string | null | undefined` via its internal `toMs` helper. The coercion is harmless but unnecessary — a minor readability debt.

### Summary

PR #319 is a clean, minimal fix. The root-cause diagnosis (`slaWorkingMinutes` overwrite) is correctly documented and the fix is correct. Two issues require attention before production: the missing transaction wrapper in the backfill migration (High) and the silent fallback for null-chat breaches (High, logging only). The test comment arithmetic error (Medium) is cosmetic but should be corrected to avoid reader confusion. No security issues.

## PR #320 — feat(surveys): targeted audience selection (`feat/gh-313-targeted-survey-audience`)

### Critical

None.

### High

**[`backend/src/api/trpc/routers/segment.ts:156–188` — `update` and `delete` procedures do not enforce ownership]**
`managerProcedure` only checks that the caller's role is `manager` or `admin`. The `update` and `delete` procedures accept any segment UUID and pass it directly to the service layer — they do not check that `segment.createdById === ctx.user.id`. This means any manager can rename or hard-delete any other manager's segment.

The `create` procedure correctly passes `ctx.user.id` as `createdById`. The `addChat` procedure passes `ctx.user.id` as `addedById`. But `update` and `delete` deconstruct `{ input }` without `ctx`:

```typescript
// Current (no ownership check):
.mutation(async ({ input }) => {        // ctx is absent
  return await updateSegment({ segmentId: input.id, ... });
})

// Current (no ownership check):
.mutation(async ({ input }) => {        // ctx is absent
  await deleteSegment(input.id);
})
```

The intended design (from the service-layer JSDoc) is: "for now we don't enforce per-segment ACLs at the service layer; the tRPC router layer guards with `managerProcedure`". This means ownership enforcement was intended to live in the router, but it is absent.

Fix: add an ownership check in both procedures, or pass `ctx.user.id` to the service and enforce there. For admin users you may want to bypass the check (admin can manage all segments).

```typescript
update: managerProcedure
  .input(...)
  .mutation(async ({ ctx, input }) => {
    // Fetch segment to verify ownership (admins bypass).
    const segment = await ctx.prisma.chatSegment.findUnique({
      where: { id: input.id },
      select: { createdById: true },
    });
    if (!segment) {
      throw new TRPCError({ code: 'NOT_FOUND', message: `Segment ${input.id} not found` });
    }
    if (ctx.user.role !== 'admin' && segment.createdById !== ctx.user.id) {
      throw new TRPCError({ code: 'FORBIDDEN', message: 'You do not own this segment' });
    }
    try {
      return await updateSegment({ segmentId: input.id, ... });
    } catch (err) { mapServiceError(err); }
  }),
```

Apply the same pattern to `delete`.

**[`backend/src/services/feedback/segment.service.ts:235` — `addChatToSegment` does not filter soft-deleted chats]**
The chat lookup uses `prisma.chat.findUnique({ where: { id: chatId }, select: { id: true } })` without a `deletedAt: null` filter. A soft-deleted chat (with `deleted_at IS NOT NULL`) will be found and a membership row will be created. When the audience is later expanded via `getActiveClients`, soft-deleted chats are correctly filtered by `deletedAt: null` in the `findMany` query — so the delivery will silently produce zero results for that chat without any error. This mismatch (membership row exists, delivery silently skipped) is observable only in survey analytics and could confuse admins.

Fix: add `deletedAt: null` to the chat lookup in `addChatToSegment`.

```typescript
prisma.chat.findUnique({ where: { id: chatId, deletedAt: null }, select: { id: true } }),
```

If the result is null, throw `CHAT_NOT_FOUND` with a message that includes "or is soft-deleted".

**[`backend/src/services/feedback/survey.service.ts:307–321` — segment existence check is TOCTOU-susceptible]**
`createCampaign` validates that all `segmentIds` exist in `chatSegment` (read), then writes the survey row. There is no database-level constraint between `FeedbackSurvey.audienceSegmentIds` and `ChatSegment.id` — the array column cannot have a FK referencing elements of a UUID array in PostgreSQL. If a segment is deleted between the existence check and the `feedbackSurvey.create` call, the campaign is persisted with a dangling segment ID.

The JSDoc on `deleteSegment` acknowledges this: "The segment id may still linger in historical `FeedbackSurvey.audienceSegmentIds` rows — that's intentional." The `getChatsInSegments` function is also documented to silently skip missing segments during delivery.

This is acceptable-by-design for historical records, but for a campaign that has not yet been delivered the silent-skip behavior could result in a 0-delivery campaign with no warning to the admin. The risk window is narrow (race between UI validate + submit and an admin deleting the segment), but the impact is confusing.

Recommendation: in `startSurveyDelivery`, after `getChatsInSegments` returns, log a warning (or surface via the survey status) if `clients.length === 0` and `audienceType !== 'all'`, so the scenario is observable.

### Medium

**[`backend/src/api/trpc/routers/segment.ts:345–377` — `getById` makes 2 sequential round-trips]**
`getById` first calls `ctx.prisma.chatSegment.findUnique` (with `_count` include), then separately calls `getSegmentChats(input.id)` which issues another `findMany` on `chatSegmentMember`. These two queries are sequential and could be collapsed into a single query using `include: { members: { include: { chat: { select: { id: true, title: true } } } } }`, eliminating one DB round-trip per call.

```typescript
const segment = await ctx.prisma.chatSegment.findUnique({
  where: { id: input.id },
  include: {
    createdBy: { select: { id: true, fullName: true } },
    members: {
      include: { chat: { select: { id: true, title: true } } },
      orderBy: { addedAt: 'desc' },
    },
  },
});
// members.length replaces _count
// members.map(...) replaces getSegmentChats call
```

### Better alternatives (design notes)

**`audience_chat_ids BigInt[]` vs. polymorphic join table**

The PR stores explicit chat IDs in a `BIGINT[]` column on `feedback_surveys`. This avoids a join table and keeps the delivery snapshot immutable (chats deleted after campaign creation won't silently vanish from the persisted list). The trade-off:

- Pro: Simple schema; immutable snapshot; cheap to read (no join).
- Con: No referential integrity (FK on array elements is not supported in PostgreSQL). Querying "which campaigns targeted chat X" requires an `ANY(audience_chat_ids)` array-containment scan — no B-tree index applies; a GIN index would be needed for efficient reverse lookups.
- Con: Max 500 chat IDs (Zod guard) stored in a column; works for typical deployments but is an implicit cap.

A polymorphic `survey_audience_members(survey_id, chat_id)` join table would restore FK integrity and enable indexed reverse lookups, at the cost of an additional join on every delivery expansion. For the current scale (< 1000 chats per deployment, one-shot delivery), the array approach is acceptable. If reverse-lookup queries ("show me all campaigns that targeted chat X") are needed in the future, add a GIN index:

```sql
CREATE INDEX feedback_surveys_audience_chat_ids_gin
  ON "public"."feedback_surveys" USING GIN ("audience_chat_ids");
```

The `audience_segment_ids UUID[]` column has the same trade-off but lower concern because segment IDs are UUIDs and deletion is acknowledged-by-design.

## Overall verdict

**PR #319** — Correct fix for the +0м SLA excess regression. Two High-severity issues need addressing before production deploy: the backfill migration must be wrapped in an explicit transaction, and a warning log should be added when `chat` is null for a breached request. One cosmetic test comment error should be corrected. Ready to merge after those fixes.

**PR #320** — Solid architecture, well-tested for the happy paths. One High-severity security gap requires immediate resolution: `update` and `delete` segment procedures do not check ownership, allowing any manager to modify or destroy another manager's segments. A second High finding: soft-deleted chats can be added to segments and will silently produce zero deliveries. Several Medium code-quality items (chat list cap, double round-trip in `getById`, duplicate schema, optional fields that should be required) should be addressed before production. The TOCTOU race on segment deletion is acknowledged-by-design but the zero-delivery silent failure path should be made observable via logging. Not ready to merge until the ownership check is implemented.
