# Нарушения

> Журнал зафиксированных нарушений регламента. Используйте для разбора инцидентов и работы над ошибками.

## Обзор

В отличие от [Алертов](./alerts.md), которые предупреждают о проблеме "здесь и сейчас", раздел "Нарушения" — это исторический журнал уже свершившихся фактов несоблюдения стандартов обслуживания. Это инструмент для анализа качества работы и выявления системных проблем.

## Что считается нарушением?

Система автоматически фиксирует нарушение, когда фактические показатели выходят за рамки установленных SLA (Service Level Agreement):
- **Просроченный первый ответ**: Оператор не ответил на новое сообщение клиента в течение установленного времени (например, 15 минут).
- **Просроченное решение**: Запрос не был закрыт/решен до дедлайна.
- **Длительный простой**: Запрос висит в статусе "В работе" без активности слишком долго.

## Журнал нарушений

Таблица нарушений содержит следующие данные:
- **Дата и время**: Точный момент фиксации нарушения.
- **Тип**: Какое именно правило было нарушено.
- **Ответственный**: Оператор, на котором "висел" чат или запрос в момент нарушения.
- **Длительность просрочки**: На сколько времени был превышен лимит (например, "+1ч 20мин").
- **Контекст**: Ссылка на связанный чат или запрос.

## Функции раздела

### Фильтрация
Для анализа можно фильтровать журнал:
- **По периоду**: За сегодня, вчера, неделю, месяц.
- **По типу нарушения**: Например, смотреть только просрочки ответов.
- **По сотруднику**: Чтобы оценить дисциплину конкретного оператора.

### Экспорт отчётов
Данные о нарушениях можно выгрузить в формате CSV или Excel для дальнейшей обработки или включения в KPI отчеты.

### Разбор инцидентов
Менеджер может использовать этот раздел для проведения "разбора полетов":
1. Найти нарушение.
2. Перетий в чат/запрос по ссылке.
3. Изучить историю переписки и таймлайн действий.
4. Выяснить причину (загруженность, ошибка оператора, сложный вопрос).
5. Оставить комментарий (если предусмотрено системой) или обсудить с сотрудником.
