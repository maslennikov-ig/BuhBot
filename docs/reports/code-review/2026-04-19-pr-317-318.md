## PR #318 — docs(adr): propose per-package Release Please

**Branch:** `docs/gh-310-monorepo-versioning-adr`
**Files changed:** `docs/adr/005-monorepo-versioning-strategy.md` (new, 307 lines), `docs/plans/gh-310-monorepo-versioning-plan.md` (new, 236 lines)

### Critical

**[docs/adr/005-monorepo-versioning-strategy.md:45,86,87,95,98,106,147,211,267–275] — Fundamental factual error: Release Please routes by file path, not by commit scope**

This is the most consequential factual error in the ADR. The document asserts repeatedly and centrally:

> "Release Please reads scoped commits (`feat(frontend):`, `fix(backend):`) and routes each change to the appropriate component's release PR."
> "Scoped commits are the mechanism Release Please uses to route changes to the correct component."
> "a `fix(backend):` bump increments `backend/package.json` patch and leaves `frontend/package.json` untouched."

This is **incorrect**. Release Please multi-package mode uses **file path analysis** (`CommitSplit` utility, keyed on `packagePaths`), not commit message scopes, to assign commits to packages. A `fix(backend): handle null telegramId` commit that modifies files in `frontend/` will be attributed to the `frontend` package, not to `backend`. A `feat(frontend): add dark mode` commit that modifies only root-level files will go to the root package, not to `frontend`.

**Impact:** The entire rationale for Option B as presented is built on this incorrect premise. The "minimal developer-friction" argument ("existing `feat(frontend):` scopes are already established practice — no behavior change for contributors") is false. After the migration, contributors would need to understand that the *scope in the commit message is irrelevant to routing* — only the *files they change* matter. This is a non-trivial operational difference from what the ADR describes.

Additionally, the scopes table in Step 5 of the plan (`feat(frontend): → frontend/package.json`) is operationally misleading: if a developer writes `feat(frontend):` but edits `backend/src/`, the change goes into the backend release, not the frontend one.

**Fix (required before this ADR can be approved):**

1. Replace all scope-routing language with file-path routing language throughout the ADR and plan.
2. Revise the "Pros" list for Option B: the correct pro is "commits that touch `backend/` files automatically increment `backend/package.json` regardless of scope" — which is actually stronger than scope-based routing because it is objective and not prone to human error.
3. Revise the "Cons" list: remove "Contributors must use correct commit scopes consistently" (this is not how RP works) and replace with "Commits that touch files in multiple packages are attributed to all of them, which may cause unexpected version bumps."
4. Update Step 5 (CONTRIBUTING.md scopes table) to explain that scopes affect the *changelog section label* but not *package routing*. Routing is determined by which files the commit modifies.
5. Update the post-migration verification checklist item "Three Release Please PRs appear... for any pending scoped commits" to "for any pending commits that modified files under `backend/`, `frontend/`, or the root."

### High

**[docs/adr/005-monorepo-versioning-strategy.md:211] — `component` field description is inaccurate**

The ADR states: "Note: `component` fields in each package entry control how Release Please labels the release PR and tags."

This is a partial truth that omits the actual constraint. The `component` field is used primarily by the `linked-versions` and `group-priority` plugins — it is a logical name for grouping packages, not a general label for PR/tag naming. Tag naming for multi-component repos is controlled by the `include-component-in-tag` input in the GitHub Action (defaults to `false`). Without `include-component-in-tag: true` in the workflow, Release Please will create tags like `v0.32.3` for all packages rather than `backend-v1.0.0` and `frontend-v0.15.0`, breaking the Docker tagging strategy described in Step 6.

The plan's Step 4 does not mention adding `include-component-in-tag: true` to the workflow, which means the implementation as written would not produce the `backend-v1.0.0` style tags referenced in the plan's own YAML example (`backend_tag: 'Backend release tag (e.g. backend-v1.0.0)'`).

Fix:
1. Correct the `component` field description in the ADR.
2. Add `include-component-in-tag: true` to the `release-please-action` step in the Step 4 workflow snippet.
3. Note that without this flag, all three packages will generate `v{version}` tags which collide when versions differ.

**[docs/adr/005-monorepo-versioning-strategy.md:86, docs/plans/gh-310-monorepo-versioning-plan.md:91–103] — Output variable name mismatch: `releases_created` vs `release_created`**

The ADR and plan use `backend--releases_created` and `frontend--releases_created` (plural) throughout. The release-please-action v4 README documents the output as `<path>--release_created` (singular). The current workflow on `main` uses `steps.release.outputs.releases_created` — this is the top-level aggregate key, which is a different thing. Per-path keys follow the singular form `<path>--release_created`.

If the plan is implemented using `backend--releases_created`, the condition will always evaluate to empty string (falsy) and deploy triggers for individual components will never fire.

Fix: Replace all occurrences of `backend--releases_created` with `backend--release_created` and `frontend--releases_created` with `frontend--release_created` in both the ADR and the plan. Verify against the action README before the implementation PR is merged.

### Medium

**[docs/adr/005-monorepo-versioning-strategy.md — missing section] — Changesets not considered as alternative**

The ADR evaluates three options (Option A: synchronized mirror, Option B: per-package RP, Option C: hybrid) but does not consider **Changesets** (`@changesets/cli`), the de-facto standard for per-package versioning in Node.js monorepos. Changesets is explicitly designed for the exact problem described: independent package versioning with human-authored changelogs, without requiring commit-message parsing. Given that the ADR's primary argument for Option B is "lower friction because scopes already exist" — and that argument rests on the incorrect scope-routing premise — the omission of Changesets is a gap that weakens the decision record.

Fix: Add an "Option D: Changesets" entry. The recommendation can still be Option B (if the path-routing advantage holds), but the record should show the alternative was evaluated. Key trade-offs: Changesets requires developers to run `pnpm changeset` and commit a changeset file (more explicit, but more friction), while RP with path routing is fully automatic. Documenting this trade-off makes the decision defensible.

### Summary — PR #318

**Verdict: request-changes.**

The ADR rests on a factually incorrect claim about Release Please's routing mechanism (Critical). This error is load-bearing: the "minimal friction" argument for Option B, the scopes table in the implementation plan, the post-migration checklist, and the contributor documentation guidance all follow from it. Additionally, the output variable name mismatch (`release_created` vs `releases_created`) and the missing `include-component-in-tag: true` flag (both High) mean that a developer following the plan verbatim would produce a non-functional CI workflow. The ADR should not be approved until the Critical finding is resolved; once it is, Option B may still be the correct choice — the file-path routing is arguably *better* than scope-based routing — but the rationale needs to be rebuilt on the correct foundation.
